De l’analyse “qui marche” à l’analyse qui tient dans le monde réel

Mise en perspective

Jusqu’ici, on a appris à :

  • poser une question claire
  • construire un pipeline cohérent
  • nettoyer et explorer de manière défendable

👉 En entreprise, ce n’est pas suffisant.

Une bonne analyse doit survivre au temps, aux collègues et aux décisions.

Question centrale

Comment concevoir une analyse de données
qui reste valable, compréhensible et utile
au-delà du notebook et du stagiaire qui l’a produite ?

Fil conducteur

Trois exigences (pro) :

  • Transmissible : quelqu’un d’autre peut reprendre
  • Robuste : la conclusion résiste au changement
  • Utilisable : le message passe hors de l’équipe data

👉 Ce CM parle moins de code, et plus de durabilité.

Ce qui change en situation pro

Dans un cours, on vise souvent : “le bon résultat”.

En entreprise, on vise plutôt :

un raisonnement stable + des décisions discutables + un livrable réutilisable

Bloc 1

Pourquoi une analyse “correcte” peut échouer en situation professionnelle ?

Situation réelle

Vous rendez votre analyse.

Six mois plus tard :

  • les données ont changé
  • la question a évolué
  • vous n’êtes plus là

👉 Personne n’ose toucher à votre notebook.

Problème fondamental

Ce qui échoue n’est pas le code.

Ce qui échoue, c’est la transmission du raisonnement.

Intuition clé

Une analyse professionnelle doit :

  • survivre à son auteur
  • être comprise par d’autres métiers
  • être réutilisable dans un autre contexte

👉 L’analyse devient un objet collectif.

Concept clé : analyse comme artefact décisionnel

Une analyse n’est pas un résultat,
c’est un support de décision.

👉 Elle doit expliciter ce qu’on peut conclure…
et ce qu’on ne peut pas.

Analyse et décision

👉 L’analyse n’automatise pas la décision.

Anti‑pattern professionnel

  • “On a un chiffre, donc on décide.”
  • “Le modèle a dit X.”
  • “Le graphique est clair.”

👉 Un chiffre sans hypothèses = un risque.

Ce qui manque souvent dans les analyses “correctes”

Trois éléments oubliés :

  • Contexte : pourquoi cette question maintenant ?
  • Hypothèses : ce qu’on suppose vrai
  • Limites : où ça peut casser

👉 Sans ça, l’analyse n’est pas transmissible.

Check‑list de transmission (très courte)

À minima, un autre doit pouvoir répondre à :

  • Quelle était la question exacte ?
  • Quelles données (version, période) ?
  • Quelles règles clés / paramètres ?
  • Quelle conclusion + dans quelles limites ?

Phrase à retenir

Une bonne analyse éclaire une décision,
elle ne la remplace jamais.

Bloc 2

Comment rendre une analyse robuste au changement ?

Réalité professionnelle

Dans le temps :

  • les données évoluent
  • les règles métier changent
  • les indicateurs dérivent

👉 Une analyse figée devient rapidement fausse.

Intuition

On ne rend pas une analyse robuste en la rendant plus complexe.

On la rend robuste en rendant ses hypothèses visibles et modulables.

Concept clé : scénarios et invariants

Deux niveaux à distinguer :

  • Invariants : ce qui doit toujours rester vrai
  • Scénarios : ce qui peut varier (seuils, règles, filtres)

👉 Une analyse sérieuse sépare les deux.

Exemple concret (invariants vs scénarios)

Imaginez un indicateur de “clients actifs”.

  • Invariant : “actif” = au moins 1 action mesurable
  • Scénario : fenêtre temporelle (7j / 30j / 90j)

👉 Les disputes pro viennent souvent des scénarios déguisés en invariants.

Analyse paramétrée

👉 On change les paramètres, pas la logique.

Robustesse = sensibilité (pas certitude)

Une analyse robuste ne “prouve” pas.

Elle montre que la conclusion
tient dans une plage raisonnable de scénarios.

Mini‑test de robustesse (à faire systématiquement)

  • changer 1 paramètre important
  • comparer la conclusion
  • si elle bascule, le dire

👉 Un résultat qui s’effondre au moindre réglage est fragile.

Bonne pratique clé

  • documenter les hypothèses
  • isoler les paramètres
  • comparer plusieurs scénarios

👉 Une analyse devient un outil de discussion.

Phrase à retenir

Une analyse robuste n’est pas figée :
elle est paramétrée.

Bloc 3

Comment communiquer une analyse à des non‑data scientists ?

Problème classique

  • trop de graphiques
  • trop de chiffres
  • trop de jargon

👉 Le message se perd.

Intuition

La valeur d’une analyse ne dépend pas de :

  • la sophistication du modèle
  • le volume de code

Elle dépend de la clarté du message transmis.

Concept clé : narration analytique

Communiquer une analyse, c’est raconter :

  • la question posée
  • les hypothèses retenues
  • les limites
  • les implications concrètes

👉 Pas le pipeline complet.

Format “1 slide décision” (professionnel)

Une bonne synthèse tient souvent sur :

  • 1 phrase de conclusion
  • 2–3 faits qui la soutiennent
  • 1 encadré “hypothèses & limites”
  • 1 recommandation conditionnelle

👉 Tout le reste est en annexe.

Ce qu’il faut dire explicitement

Un non‑data veut surtout savoir :

  • À quel point on peut se fier à ce résultat ?
  • Dans quelles conditions il ne tient plus ?
  • Que fait‑on ensuite ?

Analyse comme récit structuré

👉 Le code reste en annexe.

Ce qu’il faut éviter (vraiment)

  • raconter le notebook
  • “voici tous les graphes”
  • confondre précision et clarté

👉 Plus vous montrez, plus vous diluez.

Phrase à retenir

Une analyse qui ne peut pas être expliquée
simplement ne sera pas utilisée.

Démo — Livrable pro minimal

Démo — Situation

Votre responsable demande :

“J’ai 2 minutes. C’est quoi le message ?”

👉 Objectif : une synthèse décisionnelle.

Démo — Slide décision (structure)

Une “slide décision” contient :

  • Conclusion (1 phrase)
  • Faits (3 bullets max)
  • Hypothèses / limites (1 encadré)
  • Recommandation (conditionnelle)

Démo — Exemple de formulation

Conclusion : “L’indicateur X augmente sur 12 mois.”

Faits :

  • hausse régulière sur 4 trimestres
  • tendance stable sur plusieurs scénarios
  • pas de rupture détectée dans les données

Limites : “Résultat sensible à la définition de X (fenêtre 30j vs 90j).”

Démo — Recommandation conditionnelle

Une recommandation pro ressemble à :

“Si l’objectif est Y, alors on recommande A.
Si la contrainte Z s’impose, alors on recommande B.”

👉 L’analyse outille une décision : elle n’impose pas.

Exemple — Mesurer une variabilité

On observe une moyenne…
mais une moyenne seule peut être trompeuse.

  • deux groupes peuvent avoir la même moyenne
  • mais des dispersions très différentes

👉 L’écart-type sert à répondre à une question simple :

À quoi sert l’écart-type ?

À mesurer à quel point les valeurs sont dispersées autour d’une moyenne.

👉 Il ne “corrige” pas la moyenne, il en donne le contexte.

Décision permise / interdite

  • écart-type faible → moyenne représentative
  • écart-type élevé → moyenne fragile

👉 La question n’est pas “comment le calculer”,
mais “peut-on raisonnablement résumer par une moyenne ?”

Conclusion

Synthèse

  • Une analyse est un artefact collectif
  • Elle doit être robuste au changement
  • Elle doit être compréhensible hors du notebook

Règle d’or (prospective)

Une bonne analyse ne répond pas seulement à la question d’aujourd’hui :
elle prépare les questions de demain.

Et après ?

Dans la suite :

  • transformer une analyse en livrable pro
  • séparer notebook / rapport / décision
  • simuler des revues critiques (“et si… ?”)

👉 Objectif : penser comme un data scientist professionnel, pas comme un exécutant.
